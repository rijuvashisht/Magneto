# Getting Started

Welcome to **Magneto AI** — the multi-agent AI control plane that ships features 3–5× faster, catches security issues before they reach production, and keeps your source code under your control.

## Quick Start (60 seconds)

```bash
# 1. Install globally
npm install -g magneto-ai

# 2. Initialize — auto-detects your stack and prompts for SDD framework
cd your-project
magneto init                                    # interactive
magneto init --with typescript nextjs           # Next.js project
magneto init --with python fastapi --sdd openspec   # FastAPI + OpenSpec (brownfield)
magneto init --with java spring-boot --sdd bmad     # Spring Boot + BMAD (audit trail)
magneto init --with aws --sdd speckit               # Terraform + Spec Kit (greenfield)
magneto init --no-sdd                           # skip SDD scaffolding

# 3. Plan a task (write a .md task file first, or use an example)
magneto plan tasks/my-task.md

# 4. Run with streaming output, sandboxed for safety
magneto sandbox run --profile standard -- magneto run tasks/my-task.md --stream

# 5. Reconcile spec ↔ code drift before merge
magneto sdd sync
```

---

## What Magneto Actually Does

Raw AI assistants load whatever files look relevant and ask you to clarify. Magneto **classifies** your task, **scopes** exactly the files that matter, runs **3–4 specialised agents in parallel**, checks **security guardrails before touching anything**, and returns a merged, deduplicated report.

| Without Magneto | With Magneto |
|---|---|
| 15–40 files loaded (shotgun) | 4–8 files (task-classified) |
| 5–10 back-and-forth exchanges | 0 — all context pre-scoped |
| Re-explains project every session | Persistent `.magneto/memory/` |
| 1 generic agent pass | 3–4 parallel role-focused agents |
| No security check | Every task evaluated before execution |

## Step 2 — Write a Task File

Tasks are plain Markdown (`.md`), YAML (`.yaml`), or JSON (`.json`). The simplest form:

```markdown
# Add analytics dashboard

## Description
Add a `/dashboard` page showing daily active users, revenue, and error rate.
Uses the existing `/api/metrics` endpoint.

## Acceptance Criteria
- [ ] Page renders server-side
- [ ] Charts use Recharts
- [ ] All numbers have loading skeletons
- [ ] Unit tests for the data-fetch hook
```

Save it as `tasks/add-dashboard.md`.

## Step 3 — Plan

```bash
magneto plan tasks/add-dashboard.md
```

Magneto classifies the task, evaluates security risk, and decomposes into parallel subtasks:

```

[magneto]   1. Design /api/metrics response shape    → backend
[magneto]   2. Build dashboard page + components     → frontend
[magneto]   3. Write unit + integration tests        → tester
[magneto] ✓ Plan saved to .magneto/cache/plan-add-dashboard.json
```

### Execution modes

Control how much authority Magneto has:

| Mode | What it does |
|---|---|
| `--mode observe` | Read-only analysis — no files touched |
| `--mode assist` | Suggestions only — you apply changes *(default)* |
| `--mode execute` | Writes files with guardrails |
| `--mode restricted` | Every action requires explicit approval |

```bash
magneto run tasks/audit.md --mode observe          # safe read-only scan
magneto run tasks/refactor.md --mode execute       # automated with guardrails
magneto run tasks/deploy.md --mode restricted      # approve every step
magneto run tasks/feature.md --approve-each        # interactive per-step
```

## Choosing a Runner

Magneto has six execution backends. It auto-selects based on your environment — or you override with `--runner`.

| Runner | When to use | Data egress | Requires |
|---|---|---|---|
| `openai` | Default cloud runner | API call to OpenAI | `OPENAI_API_KEY` |
| `copilot-local` | GitHub Copilot in your IDE | None | Copilot active |
| `copilot-cloud` | Copilot Cloud endpoint | API call | `MAGNETO_COPILOT_CLOUD_TOKEN` |
| `cascade` | Windsurf / Cascade IDE | None | Windsurf active |
| `gemini` | Google AI models | API call to Google | `GEMINI_API_KEY` |
| `ollama` | Air-gapped / regulated environments | **None** | `ollama serve` running |

### Ollama — zero egress, no API key

For healthcare, finance, or classified environments where source code cannot leave your machine:

```bash
# One-time setup
ollama pull qwen2.5-coder
ollama serve

# Tell Magneto to use Ollama
export OLLAMA_HOST=http://localhost:11434
export OLLAMA_MODEL=qwen2.5-coder

magneto run tasks/audit.md --runner ollama --stream
```

```
[magneto] Runner:      ollama (qwen2.5-coder @ localhost)
[magneto] Data egress: none ✓ — all processing is local
[magneto] ✓ Task complete  metadata.dataEgress = "none"
```

Every result is tagged with `dataEgress: "none"` — verifiable in `.magneto/audit/approvals.json`.

## Power Packs — Domain Intelligence

Power packs activate stack-specific checks. Each pack ships with real rules, not just suggestions.

### Language packs

| Pack | Checks | What it finds |
|---|---|---|
| `typescript` | 8 | Import cycles, `any` leakage, unsafe casts, missing return types |
| `python` | 14 | Hardcoded secrets, `shell=True`, `eval`, missing timeouts, pickle risk |
| `java` | 14 | Unsafe deserialization, `catch Throwable`, mutable statics, thread safety |

### Framework packs

| Pack | Checks | What it finds |
|---|---|---|
| `nextjs` | 10 | Missing `'use client'`, hydration violations, server/client boundary bugs |
| `fastapi` | 10 | CORS wildcard + credentials, `debug=True`, missing Pydantic validation |
| `spring-boot` | 12 | `open-in-view=true` (N+1), `@Transactional` on private, actuator wildcard |

### Cloud / infra packs

| Pack | Checks | What it finds |
|---|---|---|
| `aws` | 16 | IAM `*`, public S3 ACL, SSH 0.0.0.0/0, unencrypted RDS, hardcoded AKIA keys |
| `azure` | ~12 | RBAC over-permission, public storage, unencrypted disks, NSG wide-open rules |

```bash
# Combine freely
magneto init --with typescript nextjs python fastapi aws
# → 58 checks active across your full stack
```

## Spec-Driven Development

Magneto ships pluggable support for the three leading SDD frameworks. `magneto init` prompts you to choose; on existing repos the framework is auto-detected.

| Framework | Best for | What it scaffolds |
|---|---|---|
| **OpenSpec** *(default)* | Brownfield / existing code | `openspec/{project.md, specs/, changes/<name>/{proposal,design,tasks}.md}` — delta-based |
| **Spec Kit** | Greenfield projects | `.specify/constitution.md` + `specs/<slug>/{spec,plan,tasks}.md` — branch-per-spec |
| **BMAD-METHOD** | Regulated / SOC2 audit | `bmad-core/agents/{analyst,pm,architect,scrum-master,developer,qa}.md` + versioned PRDs |

```bash
magneto sdd init                              # interactive prompt
magneto sdd new add-dark-mode "Theme toggle"  # scaffold a change
magneto sdd status                            # show active framework + changes
magneto sdd sync                              # reconcile spec ↔ code drift (CI exits 1 on drift)
```

The default constitution template enforces a **WHY → WHAT → HOW** rule format because LLMs ignore terse prohibitions. The reconciler statically detects three drift kinds without an LLM call: `spec-only` (specs reference missing files), `code-undocumented` (`src/` subtrees with no spec coverage), `mismatch` (tasks marked done that reference missing files).

## Sandbox & Isolation

Run Magneto, OpenClaw, and AI-generated code inside an OS-level sandbox with auto-detected runtime and graceful fallback.

| Profile | Use case |
|---|---|
| `strict` | Read-only audits — no writes, allowlist network |
| `standard` | Default for `execute` mode — RW project, denied system paths |
| `dev` | Local dev — open network, RW project |
| `off` | Trusted CI only — full host access |

**Runtimes**: Docker, Podman, macOS `sandbox-exec`, Linux `bwrap`, **Windows Sandbox** (`.wsb`), and **WSL2**.

```bash
magneto sandbox status                                      # detected runtimes + image status
magneto sandbox build                                       # build the magneto-sandbox container
magneto sandbox run --profile strict -- magneto security audit
magneto sandbox shell --profile dev                         # interactive shell
magneto sandbox doctor                                      # validate setup
```

## Zero-Trust Memory Lock

`.magneto/memory/` carries cross-session agent context. Tampering with it can poison every future agent run. Magneto locks it cryptographically:

```bash
magneto memory lock                          # SHA-256 each file, sign manifest with HMAC
magneto memory lock --require-root           # only root may unlock
magneto memory verify                        # exit 1 on tamper, missing files, or unrecorded files
magneto memory unlock                        # owner unlocks; offline-only by default
magneto memory status
```

- HMAC-SHA256 manifest signature with a key derived from `~/.magneto-key + hostname + uid` — backups don't restore across machines (intentional).
- Runtime active-task gating: while a task is running, even the owner cannot modify memory.
- Offline-only mutation: `unlock` refuses when a network interface is up unless `--allow-online` is passed (audited).

## AI Assistant Setup

Magneto governs every major AI coding assistant. After `magneto init`, the right files (skills, MCP configs, steering rules, `AGENTS.md`) are scaffolded automatically.

| Assistant | What gets written | Runner |
|---|---|---|
| Claude Code | `.claude/skills/magneto/SKILL.md` + MCP server | — |
| GitHub Copilot | `.github/copilot-instructions.md` + `.github/agents/` + `.vscode/mcp.json` | `copilot-local` / `copilot-cloud` |
| Cursor | `.cursor/rules/magneto.mdc` | — |
| Windsurf / Cascade | `.windsurf/workflows/*.md` + MCP | `cascade` |
| Google Antigravity | `~/.antigravity/skills/magneto/` + MCP | — |
| Gemini CLI | MCP-compatible auto-registration | `gemini` |
| OpenClaw | ClawHub plugin (`magneto adapter install openclaw`) | — |
| Aider / Trae / Hermes | `AGENTS.md` + skill files | — |
| Kiro | `.kiro/skills/magneto/SKILL.md` + steering | — |
| OpenAI / Codex | API runner | `openai` |
| Ollama (zero-egress) | Local API | `ollama` |
| Manus | API adapter (`magneto adapter install manus --api-key <key>`) | — |
| OpenCode | Use `magneto generate <task>` for a pre-scoped prompt | — |

**Most-secure setup** — combines the best of all features:

```bash
magneto memory lock --require-root
magneto sandbox run --profile strict -- \
  magneto run tasks/audit.md --runner ollama
```

Source code never leaves the machine, the agent process is filesystem-isolated, agent memory cannot be silently mutated, and AI output is scanned with `magneto security audit` before merge.
